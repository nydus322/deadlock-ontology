# Deadlock KV3 Wheat-vs-Chaff Heuristics

Apply these rules when classifying keys in extracted Source 2 KV3 files. Each key gets one of three labels: **wheat** (game-meaningful, include in ontology), **chaff** (engine plumbing, drop), or **unknown** (quarantine, revisit later).

## H1 — The Player Observability Test
A key is wheat only if changing its value would produce a difference a player could observe in-game. Cooldowns, damage, scaling, durations, target filters, status effects: wheat. Class IDs, resource handles, schema versions, editor flags, file paths, GUIDs, parent class names: chaff. Tiebreaker: would a wiki article mention it? If no, drop it.

## H2 — Naming Conventions Leak Intent
Source 2 uses Hungarian-style prefixes after `m_`. Type prefixes that signal **chaff by default**: `m_h` (handle), `m_p` (pointer), anything with `Editor`, `Tool`, `Preview`, `Debug`, `Gizmo`, `Spew`. Type prefixes that signal **likely wheat**: `m_fl` (float), `m_n` (int), `m_b` (bool), `m_str`/`m_psz` (string), `m_vec` (vector). Keys containing `Tooltip`, `Display`, `Localized`, or `UI` are presentation — keep but model separately from mechanical facts. `_class` keys and `CCitadel...` type values are usually engine-level, but the *fact* of the class is your type discriminator (see H4).

## H3 — Repetition Frequency Analysis
For every key path, count distinct values across all entities of the same class. **One distinct value across all instances → almost certainly chaff** (engine default). **Two distinct values → often a meaningful boolean.** **Many distinct values → almost certainly wheat** (a designed dimension of variation). Run this analysis mechanically before reading files by hand; it surfaces the ontology shape faster than intuition.

## H4 — Type Discriminator First
Every entity has a class name (e.g. `CCitadel_Ability_Abrams_1`) or `_class` key. Group files by class **before** classifying keys. Then perform a two-level diff: keys that vary *within* a class are parameters of that ability type; keys constant within a class but varying *across* classes are structural differences between types. This two-level view prevents both over-generalization and over-specialization in your taxonomy.

## H5 — Localization Tokens Are Ground Truth
Any value matching `#citadel_...` or `#ability_...` is a localization key, which proves the field is shown to players. **Every field with a localization token is wheat by definition.** Bootstrap your minimum viable ontology from exactly this set, then expand outward to numeric/mechanical fields that feed into those displayed values.

## H6 — Quarantine, Don't Delete
Maintain three buckets: wheat, chaff, **unknown**. Unknown keys go to `unknowns.jsonl` with their value samples and frequency counts. Revisit weekly. Premature deletion loses the discovery trail; quarantine costs nothing and preserves optionality.

## Output Artifact
The deliverable of applying these heuristics is a written rubric — a markdown file listing "always keep," "always drop," and "context-dependent" keys with one-line justifications each. This rubric is the foundation of your ontology and is more important than any code produced in this phase.


The first and most important heuristic from Allemang is what he calls "AAA: Anyone can say Anything about Any topic." This sounds like a slogan but it has a concrete implication at extraction time: do not treat your VPK source as the single source of truth whose structure your ontology must mirror one-to-one. The VPK is one speaker making claims about the Deadlock domain. Your ontology should be designed so that later, when you add patch notes, wiki data, community stats, or a second game version, those new speakers can add triples about the same entities without you having to restructure anything. Practically, this means when you extract a hero or an ability from the VPK, you mint a stable IRI for it that is not tied to the VPK's internal file path or key name. The VPK's heroes/hero_abrams.vdata becomes something like nydus:hero/Abrams, and the fact that it came from that file becomes a separate triple (nydus:hero/Abrams prov:wasDerivedFrom "heroes/hero_abrams.vdata") rather than being baked into the identifier itself.
The second heuristic, closely related, is "the URI is the thing, not the label." Allemang hammers this because new ontologists constantly conflate a thing's name with its identity. In VPK extraction this shows up when you're tempted to use localization keys or display strings as identifiers. Resist that. The localization string "#hero_abrams" is a property of the hero — specifically a label, probably typed with rdfs:label and tagged with a language — not the hero's identity. If Valve renames Abrams tomorrow, the IRI should not change. This discipline pays off enormously later when you're joining data across patches.
The third heuristic — and this is where the SHACL-primary mindset really diverges from the OWL-traditional one — is "model what you need to validate, not what you think is metaphysically true." Irene Polikoff's TopQuadrant posts make this point repeatedly. In an OWL-first world people agonize over whether an ability "is-a" item or "has-a" cooldown or whether cooldowns are really durations or really events. In a SHACL-first world you ask a simpler, more grounded question: what shape does a valid ability instance need to have for my downstream consumers to use it? If every ability must have a name, a cooldown, a cost, and at least one effect, then that's your property shape, and you don't need to resolve deep ontological debates about what an ability "really is." At extraction time this means you should be writing SHACL shapes in parallel with your extractor, not after — the shapes become the contract that tells you when the extractor is done and correct.
The fourth heuristic is Allemang's "distinguish the schema from the data, but keep them in the same graph." When you extract from the VPK you will find yourself pulling two very different kinds of information. Some of it is type-level: the fact that Deadlock has a concept of "ability slot," that abilities have cooldowns, that heroes have a primary stat. Some of it is instance-level: Abrams's specific abilities, their specific cooldown values at level 1. Resist the urge to pull only the instance data and leave the schema implicit in your Python extractor. Every type-level fact you discover while reading the VPK (every enum, every implicit category, every "this field is always present when that field is") should become an explicit triple or SHACL shape in your ontology. Otherwise that knowledge lives only in your extractor code and dies when you refactor.
The fifth heuristic is what Allemang calls "the open world is your friend, not your enemy" — but it has a sharp edge at extraction time. The VPK will have missing fields, fields that are present but empty, fields that default to sentinel values like -1 or "", and fields that exist for some entity types but not others. The open-world assumption says absence of a triple means "we don't know," not "false." So your extractor should not emit triples for missing or sentinel values. Emitting nydus:hero/Abrams nydus:hasCooldown -1 is actively harmful — it asserts a false fact into the graph. Emitting nothing, and letting a SHACL shape flag the missing value as a validation error when appropriate, is the correct move. The discipline is: only extract what the source positively asserts, and let shapes decide what counts as missing.
The sixth heuristic — and this one is more from Polikoff than Allemang — is "reuse vocabulary before inventing it." Before you mint nydus:hasName, check whether rdfs:label, skos:prefLabel, or schema:name already does what you need. Before you mint nydus:derivedFrom, look at PROV-O (prov:wasDerivedFrom). Before you invent a typing system for damage types, see if SKOS concept schemes fit. This matters especially at extraction time because every custom predicate you mint is a predicate you'll have to document, shape, and defend forever. The VPK will tempt you to mint dozens of predicates that mirror its internal key names one-to-one; most of those should collapse into a smaller set of reused standard predicates plus a few domain-specific ones.
The seventh heuristic is "round-trip early." Take a single hero — say Abrams — extract him end-to-end into Turtle, load him into a triplestore, write three or four SPARQL queries you actually care about ("list all abilities with cooldown under 10 seconds," "find all heroes whose ultimate does area damage"), and run them. Do this before you extract the second hero. The queries will immediately reveal modeling mistakes that would be excruciating to fix after you've extracted the full roster. Allemang implicitly teaches this by structuring his book around worked examples; Polikoff states it explicitly when she talks about iterative SHACL development. The anti-pattern is extracting everything, then querying, then discovering that your ability-to-hero relationship is backwards or that your damage values aren't queryable because you stored them as strings.
An eighth heuristic worth internalizing specifically for a game-data extraction project: "separate the stable from the volatile." Deadlock is in active development and VPK contents change every patch. Your ontology should have a clear seam between the parts that describe Deadlock-the-game (heroes exist, abilities have cooldowns, items can be purchased) and the parts that describe Deadlock-patch-X (Abrams's Shoulder Charge has an 18-second cooldown in build 5123). The stable part is your real ontology; the volatile part is versioned instance data. When you extract, tag every volatile triple with the patch or build it came from, ideally using a named graph or a reification pattern like PROV-O's activity/entity model. You will thank yourself the first time someone asks "how has Abrams changed across the last five patches" and you can answer with a SPARQL query instead of a diff of Turtle files.
A mental exercise that might help you pressure-test your extractor design before you commit: pick the single most complex entity in the VPK — probably a hero with multiple abilities, upgrades, and stat scaling — and on paper, without writing code, sketch the triples you would want to exist for that entity after extraction. Then sketch the SHACL shapes that would validate those triples. Then ask yourself, for each triple: is this fact something the VPK positively asserts, or am I inferring it from the VPK's structure? The inferred ones are where bugs and modeling mistakes hide, and separating them consciously is worth an hour of upfront thought.
One question back to you, because it will change which heuristics I'd emphasize more: is Nydus meant to be a read-only research graph (you extract, query, analyze, publish) or an evolving knowledge base that other tools and people will contribute to over time? The answer shifts how strict you should be about PROV-O, named graphs, and vocabulary reuse — the evolving case demands much more discipline at extraction time because the cost of fixing mistakes compounds.
